Part A · the substrate

DocketClock: The Reconciled Federal Comment-Deadline Substrate

A federal-only, observation-log-backed reconciliation engine that exposes every public-comment deadline as a stable, provenance-bearing object with an honest confidence/conflict flag — sold as a B2B picks-and-shovels API/webhook to regulatory-affairs and law-firm KM desks, and architected so verticals like Watershed Watch rent it via a tag/enrichment contract without forking the core.

pragmatist 7/10trust 7.5/10substrate 6.5/10 stances proposed → critiqued → synthesized

Architecture thesis

The unit of trust is a public-comment deadline whose net-new value is reconciliation + confidence + provenance, not discovery. The decisive design treats the append-only Observation log as the spine (trust-maximalist) and the ParticipationWindow as a derived, versioned projection — never a silently-mutated truth field — but it ruthlessly defers everything that does not move the first paying B2B contract (pragmatist). The substrate is made rentable by stable OCD-IDs plus a tag/enrichment contract that lets verticals scope without touching core reconciliation (substrate stance). Three critique-killing decisions are non-negotiable from day one: (1) conflict detection compares both source dates normalized to America/New_York before flagging — the known FR-2018-27875 one-calendar-day gap is a timezone artifact, NOT a conflict, so a naive >24h-in-UTC threshold would flood the CONFLICTING bucket and destroy the product's only differentiator; (2) when confidence drops to CONFLICTING/STALE we SUPPRESS the closing-soon alert AND simultaneously FIRE a conflict notification with both source rows attached — silence is a liability for deadline-liable buyers, so suppression is never silence; (3) a Week-1 frDocNum-join-hit-rate measurement is a hard go/no-go gate with a named contingency (pivot to Regs.gov-primary with docket_id/RIN fallback) before any pipeline code is written. The MVP is the reconciliation engine + the public contract; the UI is a thin internal review console, nothing more. Federal-only is a literal 'thou shalt not' until a paying customer funds a specific jurisdiction tranche.

Canonical object — ParticipationWindow (derived projection over an append-only Observation log)

ParticipationWindow is KEPT from Codex but DEMOTED from mutable source-of-truth to a derived view re-computed from the Observation log on every reconcile (trust-maximalist inversion). A single FR notice can extend multiple dockets with different new deadlines (EPA two-actions 2025-02910), so observations relate to windows MANY-TO-MANY via an observation_targets join table — one extension observation writes updates to N windows. resolved_close_utc separated from raw per-source values (substrate) and from resolved_close_display (timezone preservation) is the core data-modeling answer to three documented edge cases at once.

Pipeline

1
0. Validate (pre-architecture gate)

Measure frDocNum join hit-rate and empirical FR-vs-Regs.gov conflict rate against the live ~1,000 open windows + Mirrulations history BEFORE writing pipeline code.

2
1. Discover

Poll FR API (keyless) every 15 min for documents with comments_close_on in the future OR action/title containing extension/correction/reopening/withdrawal keywords; poll Regs.gov v4 filter[withinCommentPeriod]=true with a lastModifiedDate change-cursor.

3
2. Observe (immutable log)

SHA-256 hash every raw payload; append one Observation row (DB trigger rejects UPDATE/DELETE); skip if hash matches latest for that (source,document_id). Parse is_extension/is_correction/is_withdrawal flags and verbatim raw_dates_text at insert.

4
3. Reconcile (confidence engine)

On new observations, load the full chain, run the deterministic JSON rulebook to produce resolved_close_utc, confidence, conflict_flags, status; write/append a ParticipationWindow version.

5
4. Enrich (tag population)

After reconciliation, pluggable enrichment workers write tags within a tenant's allowed_tag_namespaces WITHOUT touching the reconciled deadline.

6
5. Publish + gate

Expose windows via REST/RSS/ICS/CSV/webhooks; gate alert eligibility HERE, not in delivery.

7
6. Verify + follow-up

After resolved_close_utc passes, re-poll 7 days for late withdrawal/correction/extension; write an AccuracyRecord (published vs actual close); emit window.finalized when a linked final rule appears.

Source Adapters

Thin stateless fetchers, one per source, returning raw payload + metadata only (no parsing/normalization downstream). FR API (keyless), Regs.gov v4 (keyed, differential lastModifiedDate cursor), GovInfo (URL resolver only — no PDF ingestion), Mirrulations/spicy-regs (OFFLINE eval/seed only, never a live freshness source).

Immutable Observation Log

Append-only store of every raw observation with payload_hash, fetched_at, parser_version, verbatim raw_dates_text, notice-type flags. The audit spine — full replay can re-derive all windows and conflicts from observations alone.

Postgres, table-level BEFORE UPDATE/DELETE trigger raising exception; JSONB raw payload; partition by fetched_at month; index (fr_document_number, source, fetched_at DESC) and (regs_object_id, source, fetched_at DESC).

Deterministic versioned JSON rulebook (RuleBox) reading the observation chain to produce resolved_close_utc, confidence, conflict_flags, status. Eastern-date comparison; non-monotonic chain handling; Haiku only for AMBIGUOUS chain classification.

Confidence model

A deterministic, VERSIONED RuleBox rulebook runs in the reconcile stage; every confidence assignment carries a structured explanation (rules_fired, source_agreement, parse_path, staleness, conflict_ids) returned in the API. Confidence is NEVER ML-scored and NEVER LLM-scored — the LLM only classifies AMBIGUOUS extension/correction chains. The cardinal rule: a bad deadline is worse than no alert, so confidence drops suppress confident pushes — but suppression always pairs with an honest conflict/needs-verification signal, never silence. Operational HIGH is defined as measurable parameters (Eastern-date source agreement, freshness SLA by time-to-close, empty extension-chain lookback), not a marketing label, because buyers carry deadline liability.

HUC-8 is the right granularity: fine enough to be meaningful to a local coalition, coarse enough that federal dockets naming a basin or tributary system fall inside it. First basin: Chesapeake Bay (Choptank, Patuxent, Susquehanna, Potomac, James HUC-8 subbasins).

Gotchas: (1) The National Map ArcGIS REST is free/keyless but rate-limited and can be slow on complex polygon queries during interactive basin-picker sessions — cache basin_profile after first resolution; never call WBD on the alert hot path. (2) Geo-RECALL (does a docket MENTION/affect this basin?) is a text-classification problem, NOT a geo-query problem — do not conflate WBD point-in-polygon with relevance scoring; the classifier is where recall is won or lost. (3) Named-water matching alone causes BOTH false positives (a USACE 'Chesapeake Bay Harbor deepening' notice that is actually a Baltimore Harbor dredge outside the Choptank HUC-8 — the classifier sees 'Chesapeake Bay' and fires) AND false negatives (an EIS describing an upstream project by COUNTY/PROJECT name with no named tributary — the named_waters filter misses it entirely). Mitigate by running county/state intersection + project-description matching in the pre-filter, and by tuning toward recall with member 'report a missed notice' correction. (4) Large multi-state basins like Chesapeake (6 states, dozens of HUC-8 subbasins) produce a high irrelevant tail at the basin scale — scope alerting to the org's SPECIFIC HUC-8 subbasins, not the whole bay, or the volunteer org will distrust a 40%-irrelevant feed. (5) WBD boundaries are revised periodically; store wbd_version in basin_profile so a boundary change triggers an explicit re-scope rather than silent drift.

The monitor → act → see-impact loop

1
MONITOR

The basin feed surfaces all in-basin windows from three sources (DocketClock-federal, EIS adapter, Open States) by coverage tier, each carrying window_type, basin_relevance (confirmed/probable/possible), days_remaining, the verbatim resolved_close_display (never a collapsed UTC string), confidence, and govinfo_url or the EPA EIS link as the authoritative source. Lead-time reminders fire at 14d/7d/48h/same-day for Tier-1. DocketClock CONFLICTING/STALE honors conflict-dual-fire: closing-soon is SUPPRESSED and a needs-verification notification fires instead with both source observations + govinfo_url — suppression is never silence. EIS-only windows (no cross-source) carry an explicit 'EIS clock — verify at EPA for amendments' label, and any FR-detected EIS extension flips the window to CONFLICTING with a side-by-side. Tier-2 state hearings are informational with an explicit 'verify with the committee before acting' gate before any reminder.

2
UNDERSTAND

Each in-basin window gets a plain-language basin brief (Haiku, generated async at classification, regenerated on window.changed): what the rule/EIS proposes, why it matters to THIS basin (citing the classifier's matched_water_bodies/CFR-parts evidence), the operative deadline (verbatim display), how to file, and what evidence is most persuasive for this docket/EIS type. Brief is labeled with confidence and is explicitly NOT a legal opinion; links govinfo_url (federal) or the EPA EIS filing + lead-agency NEPA contact (EIS).

3
ACT

Comment composer prompts for the user's position + supporting evidence (water-quality data, site observations, scientific references, local knowledge) and pre-fills basin context (water body names, HUC codes, impact language from the brief). For eligible federal Regs.gov windows: structures the draft and files via the Regulations.gov submitter API IF the Week-1 spike confirms it is open; otherwise degrades to draft + copy-paste-into-portal with a manual filed-confirmation. For EIS windows: directs to the lead agency's stated NEPA submission channel (email/portal) — and the UI states UP FRONT that EIS filing produces a second-class link-out receipt, not a verified Regs.gov submission ID. Drafts saved locally before submission; org admins can see member drafts to coordinate a coalition response before individual filing.

4
SEE IMPACT (honest procedural receipts only)

On filing: receipt shows filed timestamp, Regs.gov submission ID (where applicable; absent for EIS link-outs — stated honestly), period_open_at_filing (boolean vs resolved_close_utc), comment_count_at_filing (totalNumComments), and — when a final rule links back — a draft-to-final linkage framed strictly as 'the agency addressed these topics in the final rule that draft comments raised', never 'your comment caused this.' Org dashboard aggregates receipts for grant reporting (windows monitored, comments filed this quarter, dockets participated in). The honest frame — 'you participated, on time, in a real federal process' — holds regardless of outcome and is the receipt the dossier requires. The sideways DocketClock confidence feed (in-basin CONFLICTING windows) is licensable to newsrooms/nonprofits as a separate read-only signal (deferred until DocketClock's public /conflicts feed is stable).
